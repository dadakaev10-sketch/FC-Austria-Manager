01 Executive Summary

FC Austria Manager ist eine moderne, cloud-basierte Webapplikation zur vollständigen Verwaltung von Fussballvereinen. Die Plattform richtet sich an Jugendakademien, Amateurvereine und semi-professionelle Clubs in Österreich und dem DACH-Raum.

02 Product Vision & Ziele

Vision Statement

FC Austria Manager soll die zentrale digitale Plattform für österreichische Fussballvereine werden — von der Jugendakademie bis zur Kampfmannschaft. Die Applikation vereint Mannschaftsverwaltung, Trainingsplanung, Spielmanagement und Vereinskommunikation in einer einheitlichen, intuitiven Oberfläche.

Zielgruppen

RolleBeschreibungHauptfunktionen
AdminVereinsvorstand / IT-VerantwortlicherVollzugriff, Benutzerverwaltung, Vereinseinstellungen
ManagerSportlicher LeiterTeams, Spieler, Trainings, Spiele verwalten
CoachTrainer der MannschaftTraining planen, Anwesenheit, Spielaufstellung
Assistant CoachCo-TrainerTrainings einsehen, Anwesenheit markieren
PlayerSpieler (Jugend)Eigene Termine, Statistiken einsehen

Funktionsumfang

  • Mannschaftsverwaltung — Teams nach Alterskategorien (U8–Kampfmannschaft), Spielerzuordnung
  • Spielerverwaltung — Vollständige Profile, Statistiken (8 Attribute), Multi-Team-Zuordnung
  • Trainingsplanung — Einheiten erstellen, Anwesenheit tracken, Schwerpunkte dokumentieren
  • Spielmanagement — Ergebnisse, Ereignisse (Tore, Karten), Wettbewerbe
  • Kalender — Monats-/Wochenansicht, alle Termine aggregiert
  • Kommunikation — Ankündigungen, automatische Erinnerungen
  • Statistiken — Mannschaftsleistung, Tore-Vergleich, Spielerauswertung
  • Einstellungen — Profil, Vereinsdaten, Rollenverwaltung

Architekturprinzipien

  1. Client-Side First — Keine serverseitige Logik; das gesamte Business-Logic läuft im Browser via Firebase SDK.
  2. Real-Time by Default — Firestore onSnapshot Listener für alle Daten, sofortige UI-Updates.
  3. Multi-Tenancy via clubId — Jeder Datensatz enthält ein clubId Feld; Client filtert automatisch.
  4. Component-Driven — Wiederverwendbare UI-Komponenten mit klarer Trennung.
  5. Type-Safe — TypeScript strict mode, alle Daten-Interfaces in database.ts definiert.

04 Backend Architecture

Das Backend basiert vollständig auf Firebase (Google Cloud). Es gibt keinen eigenen Server — alle Datenoperationen geschehen direkt vom Client via Firebase SDK.

4.1 Firebase Authentication

Authentifizierung erfolgt über Email/Password via Firebase Auth. Bei der Registrierung wird automatisch ein Firestore-Profil unter users/{uid} erstellt.

// Registrierung (auth.ts)
async function signUpWithEmail(email, password, fullName) {
  const credential = await createUserWithEmailAndPassword(auth, email, password);
  // Profil in Firestore erstellen
  await setDoc(doc(db, 'users', credential.user.uid), {
    fullName,
    email: credential.user.email,
    role: 'manager',  // Standard-Rolle
    clubId: null,      // Wird beim Onboarding gesetzt
    createdAt: serverTimestamp(),
  });
}

4.2 Generic Service Factory

Alle Firestore-Collections verwenden eine generische Service-Factory, die CRUD-Operationen und Echtzeit-Subscriptions bereitstellt:

function createService<T extends { id: string }>(
  collectionName: string,
  defaultOrderField: string,
  defaultOrderDir: 'asc' | 'desc'
) {
  return {
    subscribe(callback, constraints)     // Echtzeit-Listener
    subscribeCustom(callback, constraints) // Benutzerdefinierte Queries
    create(data)                         // Dokument erstellen
    update(id, data)                     // Dokument aktualisieren
    delete(id)                           // Dokument löschen
    getById(id)                          // Einzeldokument abrufen
  };
}

4.5 Error Handling

Alle onSnapshot-Listener implementieren einen Error-Callback, der bei Fehlern (z.B. fehlende Indexes) ein leeres Array zurückgibt, damit die UI nicht im Ladezustand hängen bleibt:

onSnapshot(q,
  (snapshot) => { callback(items); },
  (error) => {
    console.error(`[Firestore] Error: ${error.message}`);
    callback([]);  // UI bekommt leeres Array statt Endlos-Loading
  }
);

8.2 Mannschaftsverwaltung (Teams)

  • Liste: Grid-Ansicht aller Teams mit Kategorie-Badge, Saison, Spieleranzahl, Trainer
  • Erstellen: Modal mit Name, Kategorie (U8–Kampfmannschaft), Saison
  • Detail: Kader-Tab mit Spielertabelle, Links zu Spielerprofilen
  • Bearbeiten/Löschen: Hover-Overlay mit Stift- und Papierkorb-Icons

8.3 Spielerverwaltung (Players)

  • Liste: Tabellenansicht mit Suche, Team-Filter, Position-Filter
  • Profil: Vollständige Spielerdaten (Körpermaße, Kontakt, Eltern-Info)
  • Statistiken: 8 Attribute auf 0–10 Skala mit visuellen Balken
  • Multi-Team: Spieler können mehreren Mannschaften zugeordnet sein

8.4 Trainingsplanung

  • Liste: Gruppiert nach Heute/Diese Woche/Vergangen
  • Erstellen: Mannschaft, Datum, Zeit, Ort, Schwerpunkt, Notizen
  • Detail: Übersicht + Anwesenheits-Tab mit Spieler-Checklist
  • Attendance: 4 Status: Anwesend, Verspätet, Verletzt, Abwesend

8.5 Spielmanagement (Matches)

  • Kommende Spiele: Gegner, Heim/Auswärts-Badge, Datum, Wettbewerb
  • Ergebnisse: Endstand mit Sieg/Unentschieden/Niederlage-Badge
  • Detail: Ergebnis bearbeiten, Spielereignisse (Tore, Karten, Wechsel) hinzufügen
  • Filtern: Nach Mannschaft und Wettbewerb

8.6 Kalender

  • Ansichten: Monats- und Wochenansicht mit Navigation
  • Datenquellen: Trainings (grün), Spiele (blau), Events (gelb)
  • Detail-Panel: Rechte Seitenleiste mit Tagesdetails
  • Lokalisierung: Deutsche Wochentage und Datumsformate

8.7 Kommunikation

  • Ankündigungen: CRUD mit Typ (Allgemein, Training, Spiel, Eltern), Pin-Funktion
  • Erinnerungen: Automatisch generiert aus kommenden Trainings und Spielen

8.8 Statistiken (Analytics)

  • Mannschafts-Tab: Gesamt-Statistiken, Siege/Unentschieden/Niederlagen, Tore-Vergleich (BarChart)
  • Spieler-Tab: Top-Torschützen und Assists (geplant über match_events)

8.9 Einstellungen

  • Profil: Name, E-Mail (readonly), Telefon, Avatar
  • Verein: Name, Adresse, Stadt, Land, Website (Admin/Manager)
  • Benutzer: Rollenverwaltung per Dropdown (nur Admin)
  • Benachrichtigungen: Toggle für Training/Spiel/Ankündigungen

10 User Manual / Bedienungsanleitung

10.1 Erste Schritte

Schritt 1: Registrierung

  1. Öffne fc-austria-manager.vercel.app
  2. Klicke auf "Registrieren"
  3. Gib deinen Namen, E-Mail und Passwort (min. 6 Zeichen) ein
  4. Klicke "Konto erstellen"

Schritt 2: Verein anlegen

  1. Du wirst automatisch zur Onboarding-Seite weitergeleitet
  2. Gib den Vereinsnamen ein (z.B. "FC Austria Wien Jugend")
  3. Optional: Stadt und Land eingeben
  4. Klicke "Verein erstellen"
  5. Du bist jetzt Admin deines Vereins
🖼
Screenshot: Onboarding-Seite
Vereinserstellung mit Name, Stadt und Land

Schritt 3: Mannschaft erstellen

  1. Gehe zu "Teams" in der Sidebar
  2. Klicke "Team erstellen"
  3. Wähle Name, Kategorie (z.B. U15) und Saison
  4. Klicke "Speichern"

Schritt 4: Spieler hinzufügen

  1. Gehe zu "Spieler" in der Sidebar
  2. Klicke "Spieler hinzufügen"
  3. Fülle Name, Position, Rückennummer aus
  4. Wähle eine oder mehrere Mannschaften
  5. Optional: Kontaktdaten und Eltern-Info
  6. Klicke "Speichern"

10.2 Training planen

  1. Gehe zu "Trainings"
  2. Klicke "Training erstellen"
  3. Wähle Mannschaft, Datum, Start-/Endzeit
  4. Gib Ort und Schwerpunkt ein (z.B. "Passspiel")
  5. Klicke "Speichern"
  6. Das Training erscheint automatisch im Kalender

10.3 Anwesenheit tracken

  1. Gehe zu "Trainings" und klicke auf ein Training
  2. Wechsle zum "Anwesenheit"-Tab
  3. Für jeden Spieler den Status setzen: Anwesend / Verspätet / Verletzt / Abwesend
  4. Die Daten werden automatisch in Firestore gespeichert

10.4 Spiel eintragen

  1. Gehe zu "Spiele"
  2. Klicke "Spiel hinzufügen"
  3. Wähle Mannschaft und gib Gegner ein
  4. Setze Datum, Anstosszeit, Spielort
  5. Wähle Heim/Auswärts und Wettbewerb
  6. Klicke "Speichern"

10.5 Ergebnis & Spielereignisse

  1. Klicke auf ein Spiel, um die Detailseite zu öffnen

11.3 Firestore Security Rules

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // Benutzer: nur eigenes Profil schreiben
    match /users/{userId} {
      allow read: if request.auth != null;
      allow create: if request.auth != null && request.auth.uid == userId;
      allow update: if request.auth != null && request.auth.uid == userId;
    }
    // Alle anderen Collections: authentifiziert = Zugriff
    match /teams/{docId} { allow read, write: if request.auth != null; }
    match /players/{docId} { allow read, write: if request.auth != null; }
    // ... (analog fuer alle 10 weiteren Collections)
  }
}

11.4 Datenisolierung

Aktuelle Implementierung Die Datenisolierung zwischen Vereinen erfolgt clientseitig über clubId-Filterung. Firestore Rules erlauben jedem authentifizierten Benutzer Lese-/Schreibzugriff auf alle Collections. Für Produktionsbetrieb wird empfohlen, serverseitige clubId-Validierung in den Firestore Rules zu implementieren.

12.2 Skalierungsstrategie

  1. Phase 1 (Aktuell): Single Firebase Project, Client-Side Rendering, Vercel Hosting
  2. Phase 2 (100+ Vereine): Firebase Security Rules verschärfen, Pagination für Listen, Firebase App Check
  3. Phase 3 (1000+ Vereine): Cloud Functions für Business Logic, Firestore Sharding, Caching Layer
  4. Phase 4 (Enterprise): Multi-Region Deployment, Dedicated Firebase Projects, Admin Dashboard

12.3 Performance-Optimierungen

  • Turbopack — Schnellerer Build als Webpack
  • Static Generation — 13 von 18 Seiten statisch vorgerendert
  • Real-Time Subscriptions — Nur geänderte Dokumente werden übertragen (delta sync)
  • Zustand — Minimale Re-Renders durch selektive State-Updates
  • Composite Indexes — 11 Indexes für optimierte Queries
